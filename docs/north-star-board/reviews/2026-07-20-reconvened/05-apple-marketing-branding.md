# Apple Marketing And Branding Seat

## 1. Verdict

**Revise the founding verdict and replace the milestone framing.**

The engine and artifact value remain credible and sustained real use is established. What remains unproven is larger than low-ceremony ingestion: Meeting Ingest cannot yet provide trustworthy continuity or identify the approved logic that will run next (`00-source-brief.md:17-23`, `00-source-brief.md:59-63`).

Replace **Just Works Ingest** with **Just Works Continuity**, preserving one-host and one-request proof while requiring build certainty, next-meeting success, and privacy-safe treatment of accumulated history. Current posture remains internal/private alpha for the maintainer.

## 2. What The New Evidence Changes

The product category changes from transcript utility to meeting-continuity system. “Ingest” is the entry action; trustworthy project memory over time is the benefit. The first audience is now clear: the maintainer in one named agent host, later similarly technical owners. HTV establishes sustained value but not current-contract adoption; Spelman provides failure evidence, not scale; the six local copies provide no independent proof (`00-source-brief.md:145-157`). Build identity becomes part of the product promise rather than release plumbing.

## 3. Evidence

- HTV's 190 Markdown files and substantial ledger, signal, archive, and done history demonstrate sustained use (`00-source-brief.md:86-96`).
- Only 37 HTV artifacts declare current schema/transcript boundaries, while 81 ledger and 81 signal records are legacy-invalid (`00-source-brief.md:98-122`).
- The engine's deterministic capabilities and 231 tests establish technical credibility (`00-source-brief.md:159-173`).
- Original language already emphasizes durable knowledge and a strong done process (`README.md:3-21`).
- Strong artifacts function as project operating records (`docs/current-output-evaluation.md:22-37`).
- The normal session workflow still contains internal request, extraction, and completion stages (`docs/session-provider-inbox-agent-workflow.md:38-48`).
- Package, CLI, artifact, and run-summary identity are insufficient (`pyproject.toml:5-8`, `src/meeting_ingest/cli.py:17-125`, `docs/artifact-contract.md:169-220`, `src/meeting_ingest/run_summary.py:9-51`).
- The hook cannot establish consumer-visible freshness (`scripts/git-hooks/refresh-global-tool.sh:1-12`).

## 4. What Is Working

- Durable structured meeting knowledge is a coherent, demonstrated promise.
- Strong done process, deterministic behavior, human-readable artifacts, and summary-plus-verbatim are defensible.
- HTV is valid authoritative dogfood when described as a mixed-generation corpus.
- Model-assisted extraction with engine-owned side effects is sound.
- No issue was found with retaining `meeting-ingest` as the alpha technical name.
- No issue was found with host neutrality as a long-term architecture principle.
- No issue was found with using HTV as the sole authoritative sustained-use source.

## 5. Gaps And Risks

- “Ingest” sounds transactional while continuity is the distinctive value.
- Naming many hosts inflates the supported audience before one host is proven (`README.md:23-35`).
- Tests, copied artifacts, legacy history, and current outputs could be blended into misleading evidence.
- Two executables can run different code under one version.
- Results cannot reconstruct their exact engine/workflow provenance.
- Hooks are not an immutable or visible release channel.
- Undifferentiated health language makes a mixed corpus look simply broken.
- Briefing has neither a current generation nor reviewed identity base in the power-user corpus.
- The product has extensive real use yet cannot explain how much history is usable.

## 6. Recommendations In Priority Order

1. Lead with: **“Turn every meeting into a trustworthy project record—and keep the history usable.”** Supporting line: **“Trust the next meeting. Keep the history.”**
2. Adopt Just Works Continuity with one host, one invocation, build certainty, next-meeting success, and read-only history reckoning.
3. Establish one canonical immutable private-alpha channel and visibly separate editable development builds.
4. Add pre-meeting identity for executable, version, immutable build, install mode, channel, dirty state, workflow contract, skill compatibility, and update status.
5. Record that identity in runs, artifacts, handoffs, and ledgers.
6. Produce a privacy-safe dry-run history classification.
7. Prove continuity through one fresh meeting and one representative historical slice in a real follow-on task.
8. Use precise evidence language that separates 231 tests, sustained HTV use, 37 current-schema artifacts, and unproven adoption/Briefing.
9. Keep the technical name but lead product language with continuity; revisit naming only after proof.
10. Begin demonstrations with approved-build and health proof before ingestion.

## 7. What To Stop, Defer, Retire, Or Simplify

Stop copied-sample counts, static version as release identity, hook freshness claims, editable/frozen equivalence, platform-list positioning, and aggregate corpus size as compatibility proof. Defer public launch, general self-service, extra hosts/providers/sources, Guidance, deeper iQ integration, identity expansion, and Briefing marketing. Retire the complete ignored local runtime corpus after uniqueness confirmation. Simplify health into corruption, legacy incompatibility, adoption opportunity, recoverable residue, and advisory warning.

## 8. Release Decision

Do not launch publicly or declare beta. Continue as private alpha for the maintainer, but do not promote it as an approved client-work release until one immutable build can identify itself and pass consumer-visible preflight.

Exit requires one audience, host, and channel; visible immutable build; unambiguous executable selection; engine/skill compatibility; successful next-meeting proof; reviewed dry-run history classification; a continuity task using qualified history; and accurate evidence language.

## 9. Confidence And Unresolved Questions

Confidence is high on positioning, evidence discipline, build-identity priority, and release posture. Open questions include the reference host, approved-build form, update-channel model, legacy read-only eligibility, privacy-safe continuity demonstration, whether Just Works Continuity stays a milestone name or becomes a descriptor, and how Briefing should consume qualified history.
